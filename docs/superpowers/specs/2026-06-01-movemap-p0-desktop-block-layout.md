# Movemap P0 데스크탑 블록 단위 구조 개편안

## 목적

이 문서는 Sway 캡처 비교를 바탕으로, Movemap 데스크탑 편집기를 **화면 블록 단위로 재구성**하기 위한 P0 개편안이다.

이번 문서의 목표는 스타일 제안이 아니라 아래를 명확히 하는 것이다.

- 화면을 어떤 블록으로 나눌 것인가
- 각 블록은 어떤 역할만 가져야 하는가
- 현재 어디가 섞여 있고 어디를 분리해야 하는가
- 실제 구현 순서는 어떻게 되는가

---

## 전제

이번 P0는 다음 원칙을 따른다.

1. Movemap의 핵심 정체성은 `transition-first`다.
2. Sway의 구조적 장점은 참고하되 그대로 복제하지 않는다.
3. 상단 액션 / 좌측 편집 / 중앙 무대 / 하단 타임라인 / 공유 surface를 명확히 분리한다.
4. P0에서는 새 기능 추가보다 **기존 기능 재배치**를 우선한다.

---

## 현재 문제 요약

현재 Movemap 데스크탑 편집기에는 필요한 기능이 대부분 이미 존재한다.

- 저장
- 공유
- 음악 업로드
- 무대 편집
- 2D/3D 전환
- 전환 리뷰
- 타임라인
- 선택 대형 정보

문제는 기능 부족보다 **블록 계층이 약한 것**이다.

대표 문제:
- 상단 command bar와 작업 내용의 경계가 약함
- 전환 리뷰가 중요하지만 상단에 붙어 있어 command 영역과 충돌함
- 좌측 편집 IA가 제품 언어로 고정되지 않음
- 공유가 협업 surface가 아니라 액션 메뉴처럼 보임
- 타임라인이 중요하지만 화면 중심축으로 더 강하게 읽히지 않음

---

## 목표 블록 구조

P0에서 데스크탑 편집기를 아래 5개 블록으로 나눈다.

1. **Global Command Bar**
2. **Left Work Panel**
3. **Main Stage Canvas**
4. **Bottom Timeline Rail**
5. **Right Context Surface**

---

## 전체 레이아웃 개요

```text
┌──────────────────────────────────────────────────────────────────────┐
│ 1. Global Command Bar                                               │
├───────────────┬──────────────────────────────────────┬───────────────┤
│ 2. Left Work  │ 3. Main Stage Canvas                 │ 5. Right      │
│ Panel         │                                      │ Context       │
│               │                                      │ Surface       │
├───────────────┴──────────────────────────────────────┴───────────────┤
│ 4. Bottom Timeline Rail                                             │
└──────────────────────────────────────────────────────────────────────┘
```

핵심은 다음이다.

- 상단은 **프로젝트 전역 액션만** 둔다.
- 좌측은 **편집 범주 선택과 범주별 주 작업**만 둔다.
- 중앙은 **무대와 현재 선택 상태**에 집중한다.
- 하단은 **시간축과 전환 편집 중심축**이다.
- 우측은 **선택된 대상의 상세 정보 / 공유 / 리뷰 / 경고**를 보여준다.

---

## 1. Global Command Bar

## 역할
프로젝트 전역 액션만 담당한다.

## 여기에 들어갈 것
- 프로젝트 제목
- 저장 상태
- 저장하기
- 프로젝트 메뉴
- 음악
- 공유
- 2D / 3D 보기 전환
- 로그인/계정 상태

## 여기서 빼야 할 것
- 전환 리뷰 본문
- 선택 대형 세부 정보
- 무대 도구 상세 토글
- 경로 경고 통계 본문

## 구조 제안

```text
[프로젝트명] [저장상태]

[저장] [프로젝트] [음악] [공유] [2D/3D]                [로그인/계정]
```

## 의도
- 상단을 보자마자 “이건 프로젝트 command bar구나”가 읽혀야 한다.
- 지금보다 더 얇고 더 또렷해야 한다.
- Sway에서 배울 점은 **전역 액션이 한 줄로 정리되는 구조**다.

## 구현 포인트
- `공유`, `음악`, `프로젝트`는 드롭다운/패널을 열더라도 진입 버튼은 상단 고정
- 저장 상태는 버튼들 사이에 섞기보다 제목 근처에 붙이는 편이 낫다

---

## 2. Left Work Panel

## 역할
편집 범주 선택과 범주별 핵심 작업을 담당한다.

## 대분류
- Cast
- Formation
- Timeline
- Stage
- Review & Share

## 기본 동작
- 한 번에 한 범주만 연다.
- 좌측 패널은 “현재 내가 어떤 종류의 작업을 하고 있는지”를 알려주는 역할이다.
- 상세 수치 조정까지 전부 다 넣지 말고, 자주 쓰는 핵심 제어만 둔다.

## 범주별 내용

### Cast
- 출연자 목록
- 검색
- 이름/라벨/색상
- 역할/파트너 정보
- 출연자 추가/삭제

### Formation
- 현재 대형 이름
- 대형 메모
- 선택 대상 편집
- 대형 복제/삭제 관련 진입

### Timeline
- 도착 시각
- 이동 시간
- move mode
- 전환 경고 요약 진입

### Stage
- 격자/스냅
- 기준선 표시
- 무대 크기
- 무대 보기 관련 설정

### Review & Share
- 보기 링크
- 편집 링크
- PNG/PDF export
- 리뷰용 설명

## 의도
- 현재 흩어진 기능들을 제품 언어로 묶는다.
- 사용자는 “이건 출연자”, “이건 타임라인”, “이건 무대 설정”처럼 기억하게 된다.

## 구현 포인트
- P0에서는 먼저 탭/세그먼트 구조부터 만든다.
- 세부 필드를 완벽히 옮기기보다 **범주 뼈대**를 먼저 세운다.

---

## 3. Main Stage Canvas

## 역할
현재 대형과 선택된 퍼포머/오브젝트를 조작하는 핵심 공간이다.

## 여기에 남겨야 할 것
- 무대 SVG/캔버스
- 토큰 직접 배치
- 기본 선택/드래그 상호작용
- 최소한의 상단 캔버스 툴바
- 2D/3D view body

## 여기에 과하게 두지 말아야 할 것
- 공유 액션 본문
- 출연자 편집 상세 폼
- 타임라인 세부 편집 UI
- 전환 리뷰 전체 본문

## 캔버스 상단 미니 툴바 제안
- Undo / Redo
- Snap
- Path visibility
- Stage zoom/fit
- Reference visibility

즉, 캔버스 툴바는 **무대 조작 도구만** 남긴다.

## 의도
- 지금보다 더 “메인 작업 공간”처럼 보여야 한다.
- 전환 정보는 중요하지만 캔버스 위에 계속 얹기보다 옆/아래 문맥으로 분산하는 편이 낫다.

---

## 4. Bottom Timeline Rail

## 역할
Movemap의 진짜 중심축.
시간, 순서, 이동, 전환을 한곳에서 읽히게 해야 한다.

## 여기에 있어야 할 것
- 재생/이동 컨트롤
- 현재 시간
- formation blocks
- 확대/축소
- audio row
- markers / warnings / guides 확장 여지

## 타임라인에서 더 강해져야 할 것
- 현재 선택 대형과의 연결감
- arrival / duration 가시성
- 전환 경고와의 연결
- pair/group movement 문맥

## 구조 제안

```text
[재생 컨트롤][현재시간][확대/축소]
[Forms row]
[Audio row]
[Warnings / Guides row - P1 후보]
```

## 의도
- 사용자가 이 앱을 “대형 찍는 앱”이 아니라 “시간과 이동을 설계하는 앱”으로 느끼게 만드는 핵심 블록이다.
- Sway보다 더 강한 존재감을 가져가야 한다.

## 구현 포인트
- 현재의 `전환 리뷰` 일부 내용을 타임라인 옆/우측 context surface와 연결
- 타임라인 선택이 곧 오른쪽 상세 정보 변경으로 이어지게 설계

---

## 5. Right Context Surface

## 역할
현재 선택된 대상의 상세 정보와 리뷰/공유/경고 문맥을 보여준다.

## 왜 필요한가
현재 Movemap은 상단과 중앙 위쪽에 정보가 몰리는 경향이 있다.
이걸 분산해서 상단을 얇게 만들고, 상세 정보는 오른쪽으로 빼야 한다.

## 여기에 들어갈 수 있는 내용
### 기본 상태
- 선택된 대형 정보
- 도착 시각
- 이동 시간
- 이동 모드
- 메모

### 선택 대상 상태
- 선택된 퍼포머 정보
- 파트너/페어 정보
- 정렬/복제/삭제 진입

### 리뷰 상태
- 전환 경고 요약
- 경로 필터
- 먼 이동/겹침 경고

### 공유 상태
- 보기 링크
- 편집 링크
- export 액션

## 동작 방식
- 오른쪽 패널은 고정 컬럼일 수도 있고,
- 화면 폭에 따라 drawer/inspector처럼 동작할 수도 있다.

P0 기준으로는 **고정 컬럼 inspector**가 가장 이해하기 쉽다.

## 의도
- 지금 상단에 있는 `전환 리뷰`를 이쪽으로 일부 이동시켜 command bar와 분리
- 공유도 여기에 넣으면 드롭다운 export 메뉴 느낌에서 벗어날 수 있다

---

## 전환 리뷰 재배치 제안

현재 `전환 리뷰`는 Movemap 강점이지만, 위치상 상단을 무겁게 만들 수 있다.

## P0 제안
- 상단 독립 블록을 없애거나 축소
- 핵심 수치만 작은 summary chip으로 남김
- 상세 리뷰는 오른쪽 Context Surface로 이동
- 경로 필터/경고는 타임라인 선택과 연동

## 예시
상단에는:
- `A · 도착 0:00.0 · 이동 0초`

오른쪽에는:
- 경로 필터
- 먼 이동 수
- 겹침 수
- 경로 상세 토글

이렇게 나누면 상단은 가벼워지고, 전환 정보는 더 체계적으로 읽힌다.

---

## 공유 블록 재배치 제안

현재 공유는 드롭다운 액션 묶음에 가깝다.

## P0 제안
상단 `공유` 버튼은 유지하되, 누르면 오른쪽 Context Surface가 `Review & Share` 모드로 열리게 한다.

## 그룹 구조
### 링크
- 보기 링크 만들기
- 편집 링크 만들기

### 리뷰 공유
- 모바일로 검토하기
- 링크 설명

### 내보내기
- 현재 PNG
- 전체 PNG
- PDF
- 프로젝트 파일

## 의도
- 협업용 링크와 export를 분리한다.
- Sway처럼 공유를 하나의 제품 surface로 다룬다.

---

## 오디오 블록 재배치 제안

P0에서는 상단 `음악` 진입 버튼만 정리하고,
상세 구조는 P1로 미뤄도 된다.

## P0 최소안
- 상단에 `음악` 버튼 배치
- 클릭 시 오른쪽 패널 또는 모달에서 상태 확인
  - 업로드됨 / 없음
  - 교체
  - 제거
  - 재생 소스 정보

## P1 확장
- `업로드`, `라이브러리`, `교체`, `연결 상태`를 별도 surface로 정리

---

## 화면 블록별 우선 구현 순서

## Step 1
Global Command Bar 정리
- 상단 액션 그룹화
- 저장상태 위치 정리
- 공유/음악/3D 진입 위치 고정

## Step 2
Left Work Panel 뼈대 생성
- Cast / Formation / Timeline / Stage / Review & Share 탭 도입

## Step 3
Right Context Surface 생성
- 선택 대형 정보와 전환 리뷰 이동
- 공유 surface 연결

## Step 4
Bottom Timeline Rail 강조
- 현재 선택과의 연결 강화
- 경고/리뷰와 문맥 연결

## Step 5
캔버스 툴바 슬림화
- 무대 조작 도구만 남김

---

## P0 완료 기준

P0가 끝났다고 말할 수 있으려면 아래가 보여야 한다.

1. 상단이 명확한 command bar로 읽힌다.
2. 좌측 패널이 제품 언어 기반 IA를 가진다.
3. 중앙 무대가 더 메인 작업 공간처럼 보인다.
4. 타임라인이 화면의 진짜 중심축처럼 느껴진다.
5. 전환 리뷰와 공유가 독립 surface로 정리된다.
