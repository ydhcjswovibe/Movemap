# Movemap Competitive Product Design

## 목적

이 문서는 현재 열려 있던 두 레퍼런스 서비스

- Formi Studio: `https://www.formistudio.app/232293/edit`
- Sway Formations: `https://swayformations.com/create-segment?id=6a155f67f6921c94809a9824`

를 조사해 **Movemap의 제품 설계 방향**으로 재구성한 경쟁 기반 설계서다.

핵심 목표는 다음 세 가지다.

1. 공개적으로 검증 가능한 레퍼런스 기능을 정리한다.
2. 현재 Movemap 구현 상태와 비교해 차이를 구조화한다.
3. Movemap이 어떤 제품으로 수렴해야 하는지 화면 구조, 기능 우선순위, 로드맵으로 정리한다.

---

## 조사 범위와 한계

### 1. Formi Studio

대상 URL은 `Edit Performance` 페이지였지만, 비로그인 상태에서 접근 시 실제 편집 화면이 아니라 아래 로그인 화면만 확인되었다.

- `Sign in to FORMI`
- `Continue with Google`
- `Continue with email`

따라서 **Formi Studio의 내부 편집 기능 전체는 현재 증거만으로 확정할 수 없다.**
이 문서에서 Formi는 다음 수준으로만 반영한다.

- 인증 게이트가 있는 SaaS 편집기 구조
- 퍼포먼스/프로젝트 편집 맥락이 존재한다는 점
- 실제 내부 UX는 추후 계정 접근 후 별도 리서치 필요

### 2. Sway Formations

지정된 `create-segment` URL은 비로그인 상태에서 홈으로 리다이렉트되었다. 다만 공개 랜딩/가이드/요금/템플릿/AI/CV 관련 페이지는 충분히 조사 가능했다.

즉, 이 문서의 Sway 분석은

- **로그인 전 공개 정보구조(IA)**
- **공개적으로 설명된 제품 가치와 기능 서사**
- **가입 퍼널, 팀 구조, 요금제, 템플릿/AI/모바일 리뷰 방향**

에 기반한다.

### 3. Movemap

Movemap은 리포지토리 기준으로 현재 구현을 함께 분석했다.

- `README.md`
- `SPEC.md`
- `MVP.md`
- `src/App.jsx`
- timeline / pair / share / audio 관련 모듈

따라서 이 문서는 단순 경쟁사 메모가 아니라, **현재 Movemap 코드베이스에 연결되는 제품 설계서**다.

## 레퍼런스 요약

## A. Formi Studio에서 확인 가능한 사실

현재 증거로 확정 가능한 것은 많지 않다.

- 퍼포먼스 편집 URL이 존재한다.
- 편집 경험은 로그인 이후에만 열리는 구조다.
- 인증 방식은 Google / email 기반이다.
- 즉, 공개 링크 중심의 즉시 사용 제품보다는 **계정 기반 워크스페이스형 편집기**에 가깝다.

### Movemap에 주는 시사점

- 향후 Movemap도 팀/워크스페이스/권한 체계가 필요해질 수 있다.
- 다만 현재 Movemap의 강점은 계정 없이도 빠르게 만들고 공유하는 단순성이다.
- 따라서 초중기에는 Formi처럼 강한 인증 게이트를 전면에 두기보다, **게스트 편집 + 필요 시 계정 업그레이드**가 더 적합하다.

## B. Sway Formations에서 확인 가능한 제품 구조

### 1. 제품 포지셔닝

Sway는 자신을 단순 드로잉 앱이 아니라 다음과 같이 설명한다.

- 캡틴은 데스크톱에서 포메이션을 만든다.
- 댄서는 모바일에서 위치와 업데이트를 본다.
- 스크린샷/채팅 혼선을 줄이고 하나의 기준 소스를 만든다.

즉 Sway의 핵심은 **formation editor** 자체보다
**desktop planning + mobile review + team workflow**에 있다.

### 2. 공개 정보구조(IA)

상단 주요 진입점:

- Dance formation guide
- Dance formation maker
- Dance formation templates
- AI dance formation generator
- Dance motion capture
- Formation planning guides
- Pricing
- Support and community
- Join a team

이 구조는 기능 나열이 아니라, 사용자의 문제를 따라가는 제품 퍼널이다.

- 이해하기: guide / blog
- 시작하기: maker / templates
- 확장하기: AI / motion capture
- 팀으로 쓰기: pricing / join team / community

### 3. 반복되는 CTA 구조

Sway에서 가장 자주 보이는 CTA는 다음 네 축이다.

- `Get Started` / `Create Team` / `Start planning`
- `Join a team`
- `Log in`
- 필요 시 이메일 문의

즉 퍼널이 분명하다.

1. 제품 이해
2. 팀 생성 또는 팀 참가
3. 로그인 후 실제 편집

### 4. 공개 기능 서사

Sway가 공개 페이지에서 일관되게 강조하는 기능은 다음과 같다.

- 데스크톱 스테이지 그리드에서 포메이션 작성
- named dancers
- spacing / visibility 관리
- transitions 계획
- mobile review 공유
- templates 기반 시작
- AI로 아이디어/variation 생성
- 장기적으로는 video → motion capture 분석

핵심은 **정적 그림 제작**이 아니라 **리허설 운영 워크플로**다.

### 5. 로그인 후 실제 편집기에서 검증한 기능 인벤토리

이번 조사에서는 Sway 자동화 세션으로 실제 로그인 후 `create-segment` 편집기 내부를 확인했다. 아래 내용은 추정이 아니라 **현재 계정/샘플 세그먼트에서 직접 확인한 UI 기준**이다.

#### 상단 편집 명령

- `File`
  - `Save` (현재 상태에 따라 비활성 가능)
  - `Version History`
  - `Download PDF`
- `Edit`
  - `Undo`
  - `Redo`
  - `Edit Segment`
  - `Move to...`
- `View`
  - `Flip Stage`
  - `View Transitions`
  - `Simple Mode`
  - `Video View`
- 별도 상단 액션
  - `Add Audio`
  - `Share`
  - `3D` 버튼
  - 저장 상태 표시 (`Saved`, `Auto saving in ...`)

#### 좌측 작업 패널

- `Roster`
  - 출연자 수 표시
  - `Search dancers`
  - 출연자 추가 버튼
  - 출연자별 좌표 목록 표시
  - 개별 퍼포머 상세 편집
    - 이름/라벨 입력
    - `Add to Roster`
    - 색상 팔레트
    - 문자/레터 지정
    - 도형 선택
    - `Pair`
    - `Remove`
- `Formation`
  - 현재 대형 인덱스 (`1 / 3` 형태)
  - `INFO`
  - `LOCATIONS`
  - `FORMATION NAME`
  - 대형별 색상 선택
  - `NOTES`
  - 노트 복제 / 삭제
- `Production`
  - `PROPS`
  - `CAMERAS`
  - `ADD SHAPES`
    - `Square`
    - `Rectangle`
    - `Circle`
    - `Cylinder`
  - 무대용 production item 목록
- `Templates`
  - `FORMATION GENERATION` (Beta)
  - 자연어 프롬프트 입력창
  - `Generate formation`
  - 개인 템플릿 저장/불러오기 문맥
  - `Add 4+ performers to browse formations` 안내
- `Stage`
  - 무대 크기 `WIDTH / HEIGHT` (feet)
  - `Grid lines` on/off
  - `Dot size` slider
  - 배경 이미지 업로드
  - `Snap dancers` (`Free / Grid / Half / Quarter`)
  - `Divisions`, `Subdivisions`
  - `Team visible` 접근 토글

#### 중앙 캔버스 / 하단 타임라인

- 무대 방향 라벨
  - `BACKSTAGE`
  - `OFFSTAGE RIGHT`
  - `OFFSTAGE LEFT`
- 스테이지 좌표축 / division 눈금 노출
- 현재 폼 선택 탭 (`F1`, `F2`, `F3`)
- 재생 제어
  - 이전 / 재생 / 다음
  - 전체 오디오 길이와 현재 시점 표시 (`0:00.000 / 2:40.038` 형태)
- 폼 편집 제어
  - `New`
  - `delete`
  - 옵션 버튼 (`tune`)
- 하단 row 구조
  - `Forms`
  - `Audio`
  - `Rows` 설정 버튼
- 타임라인 상에 폼 블록 추가 버튼

#### 공유 / 오디오 흐름

- `Share`
  - 기본 접근 수준이 `Restricted`
  - 옵션으로 `Anyone with the link` 존재
  - 팀 멤버만 열 수 있다는 설명 노출
  - `Get link` 버튼 제공
- `Add Audio`
  - `Upload New`
  - `From Library`
  - drag-and-drop 업로드 영역
  - 업로드 후 `Add Audio`로 세그먼트에 반영하는 흐름

#### Movemap에 주는 더 구체적인 시사점

1. **편집기 자체는 이미 Movemap과 매우 가까운 문제를 푼다.**
   - 스테이지 캔버스, 폼 시퀀스, 오디오 타이밍, 공유까지 한 화면에 묶여 있다.
2. **Movemap이 앞으로 보강해야 할 것은 패널 구조의 체계화다.**
   - 현재 Movemap 기능은 강하지만, Sway처럼 `Roster / Formation / Production / Templates / Stage`로 정보가 명확히 구획되면 학습 비용이 내려간다.
3. **템플릿과 production item은 Movemap의 차기 확장 후보로 적합하다.**
   - 특히 소품/카메라/무대도형은 공연 리허설 문맥에서 실용적이다.
4. **공유 권한 모델은 Movemap에도 필요하다.**
   - 지금의 단순 공유 링크에서 나아가 `restricted / link-public / team-only` 같은 접근 레벨이 있으면 팀 제품으로 확장하기 좋다.
5. **퍼포머 메타데이터 편집 밀도가 높다.**
   - 색상, 문자, shape, pairing 같은 속성은 대형이 커질수록 중요해진다.

## Sway 기능을 Movemap 관점으로 재해석

다음은 Sway의 공개 방향을 Movemap 설계 언어로 번역한 것이다.

| Sway 관점 | Movemap으로 번역한 요구사항 |
| --- | --- |
| Desktop to phone | 편집은 데스크톱 중심, 리뷰는 모바일 친화적이어야 함 |
| Team workflow | 프로젝트 단위가 아니라 팀/수업/공연 단위 운영 모델 필요 |
| Templates | 인원 수/무대 형태별 시작 대형 라이브러리 필요 |
| AI ideas | 완성 자동화보다 대형 후보 제안/변형 제안이 적합 |
| Mobile review | 공유 링크가 단순 보기 이상으로 재생/리뷰에 최적화되어야 함 |
| Join team | 팀 코드/초대 링크/권한 모델 필요 |
| Pricing by team | 개인 과금보다 팀 규모/운영 깊이 기준 플랜 설계가 자연스러움 |
| Motion capture vision | 장기적으로 영상 기반 동선 추출/비교 기능으로 확장 가능 |

## Movemap의 현재 제품 상태 요약

현재 코드 기준 Movemap은 이미 다음을 구현하고 있다.

### 이미 있는 것

- 프로젝트 생성
- 공연 타입 선택
- 출연자 수/이름 관리
- 무대 위 토큰 드래그 편집
- 스냅 격자
- 도착 시각 + 이동 시간 기반 대형 모델
- 하단 타임라인 편집
- 대형 추가 / 복제 / 삭제
- 이동 keyframe 편집
- 페어/파트너 묶음과 분리
- 오디오 업로드 및 재생
- 공유 링크 생성
- 읽기 전용 공유 화면
- PNG / 전체 PNG / 인쇄 PDF / 프로젝트 JSON 백업
- 로컬 저장 + Supabase 기반 클라우드 저장

### 아직 약한 것

- 제품 메시지와 정보구조
- 모바일 리뷰 전용 UX 완성도
- 팀/권한/워크스페이스 모델
- 템플릿 시스템
- AI 보조 생성
- 대규모 팀 편집 가속 도구
- 계정/플랜 모델
- 실시간 협업이나 피드백 루프

즉 Movemap은 **핵심 편집 엔진은 이미 강한 편**이고,
오히려 Sway 대비 부족한 영역은 **제품화 계층**이다.

## 벤치마크 비교 매트릭스

아래 표는 이 문서를 **실제 의사결정용 벤치 문서**로 쓰기 위해 추가한 요약이다.

태그 기준:

- `즉시 도입`: 현재 Movemap 가치와 직접 맞닿고, 기존 구조를 활용해 비교적 빠르게 반영 가능
- `중기 도입`: 필요성은 높지만 별도 화면/정책/구조 정리가 먼저 필요한 항목
- `장기 참고`: 장기 확장 가치가 있으나 MVP/근시기 핵심은 아님
- `비목표`: Sway에는 있으나 현재 Movemap 차별화 또는 우선순위와 맞지 않음

| 영역 | Sway에서 확인한 내용 | Movemap 현재 상태 | 갭 판단 | 우선순위 태그 | 구현 연결점 |
| --- | --- | --- | --- | --- | --- |
| 편집기 구조 | `Roster / Formation / Production / Templates / Stage` 패널 구조 | 핵심 기능은 있으나 패널 정보구조는 덜 선명함 | 기능보다 구조화/학습비용 개선 필요 | 즉시 도입 | 메인 편집기 contextual panel 재구성 |
| 타임라인 | `Forms / Audio / Rows`, 재생 제어, `F1/F2/F3` 블록 | formation rail, transport, range 입력 기반의 partial 상태 | 하단 타임라인을 편집 중심축으로 강화해야 함 | 즉시 도입 | `formation-rail` → bottom timeline 리디자인 |
| 음악 연동 | 오디오 길이 표시, Add Audio, 타임라인 결합 | 오디오 업로드/재생 존재 | 이미 방향은 맞고 UI 정리가 중요 | 즉시 도입 | transport + audio row 일체화 |
| 모바일 리뷰 | desktop to phone 서사, share 중심 복습 경험 | read-only 공유는 있으나 모바일 복습 UX는 약함 | 팀원이 링크만 열어도 이해되는 경험 필요 | 즉시 도입 | `/share/:id` 전용 모바일 리뷰 UI |
| 공유 권한 | `Restricted`, `Anyone with the link`, `Get link` | 공유 링크는 있으나 접근 레벨 관리 없음 | 단순 링크에서 권한 모델로 확장 필요 | 중기 도입 | share 설정/프로젝트 접근 정책 |
| 출연자 메타데이터 | 색상, 레터, shape, pair, roster 연결 | 이름/페어/파트 일부 가능 | 대형이 커질수록 메타데이터 편집성 강화 필요 | 즉시 도입 | performer detail panel 강화 |
| 템플릿 | formation generation beta, saved templates | 거의 없음 | 시작 장벽을 낮추는 핵심 영역 | 중기 도입 | 프로젝트 생성 플로우 + template picker |
| 대형 정보 | formation name, notes, locations | 타이밍/선택은 있으나 문맥 정보는 더 보강 가능 | 복습성과 편집 명확성을 높일 여지 있음 | 즉시 도입 | formation detail panel + note fields |
| 팀/워크스페이스 | team visible, join team, membership 문맥 | 없음 또는 초기 수준 | 제품화 단계에서 필요한 축 | 중기 도입 | auth / team / permission 모델 |
| Version History | File 메뉴에서 version history 제공 | 없음 | 유용하지만 핵심 편집 MVP를 막을 정도는 아님 | 장기 참고 | cloud save 이후 revision model |
| Production items | props, cameras, shapes | 없음 | 공연 운영 제품으로 확장할 때 가치 있음 | 장기 참고 | stage overlay / object layer |
| 3D / Video View | 3D 버튼, video view 노출 | 없음 | 현재 Movemap 차별화 포인트와 거리가 있음 | 비목표 | 현 단계 도입 안 함 |

### 비교 매트릭스 해석

1. **즉시 도입 영역은 “Movemap의 현재 강점”을 더 사용하기 좋게 만드는 항목**이다.
   - 타임라인, 모바일 리뷰, 출연자/대형 디테일, 편집기 구조화는 지금 코드베이스와도 가깝다.
2. **중기 도입 영역은 제품 정책이 함께 필요한 항목**이다.
   - 템플릿, 공유 권한, 팀 기능은 UI만이 아니라 저장/권한/플랜 정책이 같이 붙는다.
3. **장기 참고 영역은 벤치 대상으로는 의미 있지만 지금 당장 따라가면 집중력이 분산되는 항목**이다.
   - version history, production item은 추후 제품 확장 시점에 검토한다.
4. **비목표는 명시적으로 안 따라가는 것이 중요하다.**
   - Sway의 메뉴 밀도, 3D, video view는 현재 Movemap의 핵심 가치를 강화하지 않는다.

## 벤치 기반 우선순위 판단 기준

이 문서의 우선순위는 아래 다섯 기준으로 판단한다.

1. **핵심 사용자 가치**
   - 음악에 맞춰 빠르게 대형을 만들고 복습하게 해주는가
2. **기존 구현 활용도**
   - 현재 Movemap 코드와 자연스럽게 연결되는가
3. **학습비용 절감 효과**
   - 처음 쓰는 사용자의 막힘을 줄이는가
4. **제품화 효과**
   - 편집 엔진을 실제 제품 경험으로 끌어올리는가
5. **확장 준비도**
   - 추후 팀/플랜/AI로 확장할 기반이 되는가

이 기준으로 보면 우선순위는 다음처럼 읽는 것이 맞다.

- **최상위:** 타임라인, 모바일 리뷰, 편집기 정보구조, 디테일 패널 정리
- **그다음:** 템플릿, 공유 권한, 팀 모델
- **후순위:** production item, version history, 3D/video형 보기

## Movemap 제품 정의 재정의

## 제품 한 줄 정의

**Movemap은 음악 타이밍에 맞춰 공연 대형과 동선을 설계하고, 팀원이 모바일 링크로 쉽게 복습할 수 있게 하는 공연 리허설 맵 편집기다.**

## 핵심 가치

1. 음악에 맞춘 대형 생성이 빠르다.
2. 이전 대형을 바탕으로 다음 대형을 수정하는 흐름이 자연스럽다.
3. 이동 경로와 도착 타이밍을 함께 관리할 수 있다.
4. 팀원이 편집기가 없어도 링크로 이해할 수 있다.
5. 한국어 환경의 수업/동아리/공연 팀이 바로 쓸 수 있다.

## 차별화 포인트

Sway를 그대로 복제하지 않고 Movemap이 더 강하게 가져가야 할 포인트는 다음이다.

- 한국어 중심 워크플로
- 계정 없이도 시작 가능한 빠른 편집
- 음악 기준 `지금 시점에 대형 추가` 플로우의 단순성
- 대형 변화와 이동 시간을 한 번에 보는 편집기
- 페어/파트너, 수업형 운영, 공연팀 공유에 맞는 가벼운 협업 모델

## 핵심 사용자 플로우 설계

## 플로우 1. 빠른 대형 제작

1. 공연명과 인원 수로 프로젝트를 만든다.
2. 음악을 올린다.
3. 음악을 재생하며 대형 변경 시점에서 `대형 추가`를 누른다.
4. 새 대형은 직전 대형을 복사한 상태로 생성된다.
5. 바뀌는 출연자만 드래그 수정한다.
6. 필요 시 이동 시간을 조정하고 keyframe을 추가한다.
7. 반복한다.

이 플로우는 현재 Movemap이 가장 강한 영역이며 제품 전체의 중심으로 유지해야 한다.

## 플로우 2. 팀 공유와 복습

1. 편집자가 프로젝트를 저장한다.
2. 공유 링크를 만든다.
3. 팀원은 모바일 링크에서 현재/다음 대형과 음악 타이밍을 본다.
4. 팀원은 본인 경로나 특정 대형을 중심으로 복습한다.

현재 Movemap은 링크와 읽기 전용 모드는 있으나, **모바일 복습 전용 UX는 더 선명해질 필요가 있다.**

## 플로우 3. 템플릿에서 시작

1. 인원 수와 공연 유형을 고른다.
2. 추천 시작 대형을 선택한다.
3. 이름과 간격을 조정한다.
4. 이후 일반 편집 흐름으로 넘어간다.

현재 Movemap에는 이 플로우가 거의 없고, Sway가 제품화 관점에서 강한 지점이다.

## 플로우 4. 팀 운영

1. 팀 생성
2. 멤버 초대
3. 프로젝트별 권한 설정
4. 팀 공지/공유 링크 관리
5. 지난 프로젝트 재사용

이 플로우는 Movemap 장기 로드맵 영역이다.

Sway의 공개 설명을 참고하면 Movemap의 AI는 다음처럼 설계하는 것이 맞다.

### 하지 말아야 할 포지셔닝

- AI가 안무를 완성해 준다
- AI가 최종 대형을 결정한다

### 적합한 포지셔닝

- 현재 대형을 바탕으로 다음 대형 후보 3개 제안

- 특정 출연자 동선 과부하 경고
- spacing / visibility 관점의 추천

즉 AI는 **대체자**가 아니라 **아이디어 어시스턴트**여야 한다.

- 공유 링크
- 프로젝트 수 제한

- 권한 관리

- 공지/노트

## 2. 모바일 리뷰 화면

모바일에서는 편집보다 다음 목표가 중요하다.

- 지금 어느 대형인지
- 다음에 어디로 가는지
- 언제 움직여야 하는지
- 내 위치가 어디인지

따라서 모바일 리뷰는 편집기 축소판이 아니라 **리허설 보조 뷰**로 따로 설계하는 것이 좋다.

## 현재 Movemap과 목표 제품 간 갭 분석

### 갭 1. 제품 서사 부족
현재는 편집 기능은 있지만, 사용자에게 “왜 이 제품을 써야 하는가”를 설명하는 공개 제품 구조가 약하다.

### 갭 2. 템플릿 부재
편집 시작점이 빈 프로젝트 중심이다. 초보 사용자는 첫 배치를 만드는 것부터 막힐 수 있다.

### 갭 3. 모바일 리뷰 특화 부족
읽기 전용 링크는 있지만, 팀원이 휴대폰으로 빠르게 복습하는 경험은 더 좁고 명확하게 다듬어야 한다.

### 갭 4. 팀 모델 부재
초대/권한/참여자 워크플로가 아직 약하다.

### 갭 5. 대규모 편집 보조 기능 부족
멀티 셀렉트, 정렬, 충돌 완화, 필터링이 약하다.

### 갭 6. 제품 정책과 구현의 불일치
현재는 익명 저장 구조가 강하지만, 장기 스펙은 guest / free / paid 모델을 상정한다. 이 간극을 설계적으로 정리해야 한다.

## 우선순위 로드맵

## Phase 1. 편집기 완성도 강화

목표: 현재 핵심 편집 엔진을 실사용 수준으로 다듬는다.

- 타임라인 UX 안정화
- 대형 추가/복제/삭제 흐름 다듬기
- 멀티 셀렉트
- 정렬/간격 조정 도구
- 경로 가독성 개선
- 공유 리뷰 화면 정리

## Phase 2. 리뷰 경험 제품화

목표: 팀원이 “링크만 열어도 이해되는 경험”을 만든다.

- 모바일 리뷰 전용 UI
- 현재/다음 대형 강조
- 본인 보기 모드
- 메모/큐 표시
- 공연 진행 기준 재생 경험 고도화

목표: 새 프로젝트 시작 장벽을 낮춘다.

- 인원 수 기반 템플릿
- 무대 비율 템플릿
- 저장 템플릿
- 샘플 프로젝트 개선

## Phase 4. 팀 운영화

목표: 편집 앱에서 팀용 제품으로 올라간다.

- 로그인
- 팀 초대
- 프로젝트 목록
- 역할/권한
- 팀 단위 저장소

## Phase 5. AI / 비디오 확장

목표: 경쟁사 대비 차세대 포인트를 만든다.

- 영상 기반 참고선 추출
- 장기적으로 모션 비교 분석

## 제품 원칙

Movemap은 다음 원칙을 유지해야 한다.

1. **음악 중심**이어야 한다.
2. **대형 추가는 항상 빠르고 단순**해야 한다.
3. **화면은 편집기답게 간결**해야 한다.
4. **리뷰 링크는 비편집 사용자에게 친절**해야 한다.
5. **AI는 보조자이지 주연이 아니어야 한다.**
6. **팀 제품으로 확장하더라도 첫 사용 장벽은 낮아야 한다.**

## 최종 제안

레퍼런스를 종합하면 Movemap은 Sway처럼 “팀용 공연 운영 제품” 방향으로 갈 수 있지만, 그대로 복제하기보다 아래 전략이 더 적합하다.

### 권장 전략

- **단기:** 현재 강한 편집 엔진을 더 빠르고 명확하게 다듬는다.
- **중기:** 모바일 리뷰와 템플릿으로 제품 경험을 완성한다.
- **장기:** 팀/계정/AI/영상 분석으로 확장한다.

### 한 줄 전략 문장

**Movemap은 ‘음악에 맞춰 빠르게 대형을 만들고, 팀원이 링크로 바로 이해하는 한국형 공연 리허설 맵 툴’로 가야 한다.**
