# Sway ↔ Movemap 캡처 기반 비교표

## 목적

이 문서는 실제로 캡처한 Sway와 Movemap 화면을 기준으로,
Movemap에 가져와야 할 구조적 패턴과 그대로 두어야 할 차별점을 구분하기 위한 비교표다.

이번 비교는 감상 수준이 아니라 아래 판단을 바로 내릴 수 있게 만드는 것이 목적이다.

- 무엇이 단순 스타일 차이인가
- 무엇이 컴포넌트 수준 차이인가
- 무엇이 정보구조/편집 흐름 수준 차이인가
- 무엇을 P0로 먼저 바꿔야 하는가

---

## 캡처 범위

### Sway에서 직접 확인한 상태
- 로그인 화면
- Performances 대시보드
- 편집기 기본 상태
- Share 모달
- Add Audio 모달
- 3D 보기 상태

### Movemap에서 직접 확인한 상태
- 랜딩/프로젝트 시작 화면
- 샘플 편집기 기본 상태
- 공유 메뉴 열린 상태

---

## 비교표 읽는 법

각 항목은 아래 6개 질문으로 읽는다.

1. Sway는 이 상태를 어떻게 보여주는가
2. Movemap은 현재 어떻게 보여주는가
3. 핵심 차이는 무엇인가
4. Sway에서 그대로 가져올 점은 무엇인가
5. Movemap에 맞게 변형해서 가져와야 할 점은 무엇인가
6. 이 항목은 스타일/컴포넌트/구조 중 어디에 해당하는가

## 1. 진입/초기 제품 인상

| 항목 | Sway | Movemap | 핵심 차이 | 가져올 점 | Movemap식 변형 | 변경 수준 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 첫 진입 화면의 역할 | 공개 랜딩이 제품 가치, 팀 워크플로, 템플릿/AI/가이드 퍼널을 설명 | 프로젝트 생성 카드 중심, 빠른 시작에는 좋지만 제품 서사가 약함 | Sway는 제품 의미를 먼저 설명하고, Movemap은 바로 생성 폼으로 들어감 | 제품 가치와 사용 흐름을 초기 화면에서 더 읽히게 만드는 구성 | Movemap은 데모 체험 + 실전 편집 전환을 더 강조해야 함 | 구조 | P1 |
| CTA 위계 | 시작하기/팀 만들기/로그인/팀 참가가 명확한 퍼널 | 빈 프로젝트 시작 / 샘플로 시작으로 단순함 | Movemap은 시작은 쉽지만 계정·공유·모바일 워크플로 연결은 약함 | CTA 역할 분리 | `샘플 체험`, `실전 프로젝트 시작`, `로그인 후 저장/공유` 구조로 재편 가능 | 구조 | P2 |

### 판단
- 당장 가장 급한 건 랜딩보다 편집기 구조다.
- 다만 장기적으로는 Movemap 첫 화면도 “무대 배치 앱”이 아니라 “음악-타이밍-전환 설계 제품”으로 읽히게 바꾸는 것이 좋다.

## 2. 상단 글로벌 액션 바

| 항목 | Sway | Movemap | 핵심 차이 | 가져올 점 | Movemap식 변형 | 변경 수준 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 상단 바 역할 | File / Edit / View / Help / 3D / Add Audio / Share가 명확한 command bar를 형성 | 프로젝트명, 저장 상태, 음악 업로드, 로그인, 저장하기, 프로젝트, 공유, 도구가 있음 | Movemap도 액션은 충분하지만 command bar로서의 계층감과 분리감은 Sway보다 약함 | 전역 액션을 한 줄의 제품 command bar처럼 읽히게 하는 구조 | 메뉴형 File/Edit까지 그대로 복제할 필요는 없고, `저장·공유·3D·오디오·프로젝트 상태` 중심으로 단순화하는 편이 적합 | 구조 | P0 |
| 전역 액션과 로컬 액션 분리 | 전역 메뉴가 명확히 위에 있고, 상세 편집은 아래 패널/캔버스로 떨어짐 | 전환 리뷰가 상단 액션 바로 아래 이어져 command 영역과 작업 영역 경계가 다소 약함 | 상단의 의미가 혼합됨 | 전역 액션 바와 편집 내용 사이에 구조적 구획 삽입 | 저장상태/공유/모드전환을 상단에 고정하고, 전환 리뷰는 별도 리뷰 바 또는 inspector로 분리 | 구조 | P0 |
| 상태 인지 | 상단만 봐도 공유/오디오/3D 같은 프로젝트 확장 기능 위치가 분명 | 기능은 있으나 일부는 인풋/드롭다운/버튼으로 섞여 보임 | 액션 종류가 같은 레벨로 읽히지 않음 | 액션 종류별 분류 | `프로젝트`, `편집`, `리뷰/공유`, `확장 기능` 정도의 그룹화 추천 | 컴포넌트+구조 | P0 |

### 판단
- **가장 먼저 바꿔야 하는 영역 중 하나**.
- Movemap 상단은 기능 부족 문제가 아니라 **계층 부족 문제**에 가깝다.

## 3. 편집기의 좌/중앙/하단 구획

| 항목 | Sway | Movemap | 핵심 차이 | 가져올 점 | Movemap식 변형 | 변경 수준 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 좌측 패널 존재감 | Roster 패널이 독립된 컬럼으로 분명히 존재 | 현재 화면은 좌측 독립 컬럼보다는 상단 요약 + 중앙 편집 + 하단 타임라인 중심 | Sway는 카테고리 기반, Movemap은 흐름 기반이지만 패널 언어가 약함 | 좌측 편집 범주를 독립 컬럼으로 세우는 방식 | Movemap은 `Cast / Formation / Timeline / Stage / Review` 중 하나를 여는 구조가 적합 | 구조 | P0 |
| 중앙 캔버스 비중 | 중앙 스테이지가 메인이고 좌측/하단이 이를 보조 | Movemap도 무대 중심이지만 전환 리뷰와 상단 정보가 함께 붙어 밀도가 높음 | 무대 주변 정보가 다소 응집됨 | 메인 캔버스를 더 또렷하게 만들기 | 전환 리뷰와 선택 상세를 옆/아래 inspector로 재배치 고려 | 구조 | P1 |
| 하단 타임라인의 프레임 | Forms / Audio row가 명확히 분리돼 편집 단위가 뚜렷함 | 타임라인 기능은 있으나 전체 화면에서의 구조적 프레이밍은 Sway보다 약함 | 타임라인이 강하지만 “제품 중심축”이라는 인상은 아직 덜함 | row 구조와 타임라인 프레이밍 | Movemap은 Sway보다 더 강하게 timing/movement 경고를 타임라인과 결합해야 함 | 구조 | P0 |

### 판단
- Movemap은 이미 기능적으로 충분히 강하다.
- 문제는 **사용자가 그 기능을 어떤 구획으로 이해하는지**다.

## 4. 좌측 편집 패널 IA

| 항목 | Sway | Movemap | 핵심 차이 | 가져올 점 | Movemap식 변형 | 변경 수준 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 편집 범주 명칭 | Roster, Formation, Production, Templates, Stage처럼 이름이 바로 역할을 설명 | 현재 Movemap은 기능이 분산되어 있고, 사용자가 머릿속에 기억할 대분류 언어가 상대적으로 약함 | mental model 형성 강도 차이 | 패널을 제품 언어로 명명하는 방식 | Movemap은 `Cast`, `Formation`, `Timeline`, `Stage`, `Review & Share`가 더 적합 | 구조 | P0 |
| 출연자 편집 | Roster에서 검색/목록/추가가 한곳에 있음 | Movemap도 출연자 관리 기능은 있지만 전용 컬럼 경험은 상대적으로 덜 보임 | 출연자 관리 surface 명확성 차이 | 출연자 전용 surface | 페어/파트너/역할까지 포함한 Cast 패널로 강화 | 구조 | P1 |
| 무대/설정 편집 | Stage 문맥이 별도 범주로 존재 | Movemap은 무대 도구가 캔버스 위 툴바 중심 | 도구 접근은 빠르지만 범주 언어는 약함 | Stage 전용 범주 | 툴바는 유지하되 상세값은 Stage 패널로 빼는 혼합형이 적합 | 컴포넌트+구조 | P1 |

### 판단
- 이것도 **P0 핵심**.
- 사용자 관점에서 “어디서 무엇을 바꾸는가”를 빠르게 익히게 하려면 IA 명칭 정리가 필요하다.

## 5. 타임라인 중심성

| 항목 | Sway | Movemap | 핵심 차이 | 가져올 점 | Movemap식 변형 | 변경 수준 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 타임라인의 위치 | 화면 하단에 강하게 자리잡고 row 개념이 분명함 | 하단에 잘 위치해 있으나 상단 전환 리뷰/캔버스/설정 정보와 경쟁 | Movemap은 타임라인이 중요하지만 첫인상 중심축은 아직 분산됨 | 타임라인을 주요 편집 rail로 보이게 만드는 방식 | Movemap은 오히려 Sway보다 더 공격적으로 타임라인 중심 UI로 가야 함 | 구조 | P0 |
| 시간 정보 읽기 | 재생, 현재시간, form block 흐름이 직관적 | 현재도 충분히 읽히지만 transition-first 차별점이 더 드러날 수 있음 | 기능 부족보단 강조 부족 | 시간 축 위계 | arrival / duration / warnings / pair movement를 timeline 문맥에 더 붙여야 함 | 구조 | P0 |

### 판단
- **Movemap의 정체성은 여기서 승부**가 난다.
- Sway를 참고하되, 결과는 Sway보다 더 transition-first여야 한다.

## 종합 판단

## 그대로 배워야 할 것
1. 상단 글로벌 액션 바를 명확한 command bar로 다루는 방식
2. 편집 범주를 패널 IA 언어로 고정하는 방식
3. 공유를 별도 협업 surface로 끌어올리는 방식
4. 오디오/3D 같은 확장 기능을 문맥형 진입점으로 두는 방식
5. 화면 구획을 명확히 나눠 정보 밀도를 제어하는 방식

## Movemap식으로 더 강하게 가져가야 할 것
1. 타임라인 중심성
2. 전환 리뷰와 경고 가시성
3. pair/group movement 문맥
4. 모바일 편집을 review-only가 아니라 real editing surface로 다루는 구조

## 가져오지 말아야 할 것
1. production-management 과확장
2. mobile review-only 철학
3. transition보다 static formation 이해를 우선하는 구조

## 우선순위 제안

## P0
- 상단 command bar 재구성
- 좌측 패널 IA 재정의 (`Cast / Formation / Timeline / Stage / Review & Share`)
- 타임라인을 primary editing rail로 더 강하게 프레이밍
- 공유 surface를 드롭다운 export 메뉴에서 분리

## P1
- 전환 리뷰 위치/결합 방식 재설계
- 오디오 modal/panel 구조 설계
- 3D mode 존재감 강화
- panel density / layout hierarchy 정리

## P2
- 랜딩/초기 퍼널 개선
- 버튼 시스템/비주얼 hierarchy 정리
- 모바일 구조 전체 재설계
